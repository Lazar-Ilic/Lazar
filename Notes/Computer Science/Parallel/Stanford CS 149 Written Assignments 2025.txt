### **Direct Answer**

**2/3 exams per minute**

---

### **Explanation of Method**

In a pipeline architecture, the overall steady-state throughput is determined by its slowest component, known as the **bottleneck**. To find the bottleneck, we calculate the maximum throughput for each individual stage.

The formula for the throughput of a single stage is:


$$ \text{Throughput} = \frac{\text{Number of Workers}}{\text{Time per Task}} $$

**Step 1: Calculate throughput for each stage**

* **Stage 1 (Question 1):** Takes 3 minutes to grade, with 2 Course Assistants (CAs) working in parallel.
* Throughput = $2 / 3$ exams per minute.


* **Stage 2 (Question 2):** Takes 1 minute to grade, with 3 CAs working in parallel.
* Throughput = $3 / 1 = 3$ exams per minute.


* **Stages 3 through 7 (Questions 3-7):** Each takes 1 minute to grade, with 1 CA assigned per question.
* Throughput = $1 / 1 = 1$ exam per minute (for each stage).



**Step 2: Identify the bottleneck**
Comparing the throughputs of all stages ($2/3$, $3$, $1$, $1$, $1$, $1$, $1$), the lowest value is **$2/3$ exams per minute**.

Because the rest of the pipeline can only process exams as fast as they are handed off by the Question 1 graders, Stage 1 acts as the bottleneck and strictly limits the entire system's steady-state throughput to $2/3$ exams per minute.

Please assume that an application spawns two C++ threads that each execute the following C++ function.

void my_function(float* in, float* out, int N, int numThreads, int threadId) {

A. (10 pts) Assume you are running the application on a machine with: a single core processor that
can perform one floating point operation per clock. The core has two hardware execution contexts.
The core is connected to a memory system with infinite bandwidth, a memory READ latency of
50 cycles, a memory write latency of 0, and with a cache that has four-byte cache lines and is
16 MB in size. Cache hits have zero latency. When the application is executed on this processor
with N = 1 billion and ITERS=10, what is the steady-state utilization of the core? Please put your

"
### **Steady-State Utilization**

**33.3%** (or $\frac{1}{3}$)

---

### **Justification**

To find the steady-state utilization of the core, we need to analyze the latency of operations, the cache behavior, and how the two hardware contexts hide that latency.

* **Cache Miss Rate (100%):**
* **No Spatial Locality:** The cache line size is exactly 4 bytes, which is the exact size of a single 32-bit `float`. Fetching a cache line brings in only the single `float` being requested, meaning every subsequent load will also be a cache miss.
* **No Temporal Locality:** Each thread processes 500 million floats (2 GB) per outer loop iteration. Since 2 GB is vastly larger than the 16 MB cache, the cache is completely trashed between the `i` loop iterations.
* *Result:* Every single `in[j]` memory load is a cache miss and incurs the full **50-cycle** read latency.
* **Time per Iteration (Critical Path):**
* **Load:** 50 cycles.
* **Compute:** `ITERS = 10` iterations of `tmp *= tmp` requires 10 floating-point operations. At 1 FLOP/clock, this takes **10 cycles**.
* **Store:** **0 cycles** (given memory write latency is 0).
* *Result:* A single thread takes $50 + 10 = 60$ cycles to complete one `j` iteration. It is bottlenecked by the memory read.
* **Latency Hiding (Execution Contexts):**
* Because the core has two hardware execution contexts and infinite memory bandwidth, the two threads can issue memory requests independently and overlap their 50-cycle wait times.
* In a 60-cycle window, Thread 0 spends 50 cycles waiting for memory and 10 cycles doing computation.
* Thread 1 also spends 50 cycles waiting and 10 cycles computing within that same window.
* *Result:* The core is actively executing FLOPs for $10 + 10 = 20$ cycles out of every $60$ cycle period.

**Calculation:**

$$\text{Utilization} = \frac{\text{Compute Cycles}}{\text{Total Cycles}} = \frac{20}{60} = 33.3\%$$
"

### **The Performance Problem: Gather/Scatter vs. Contiguous Access**

The core performance issue with the original `my_ispc_function` is **strided memory access**, which prevents the hardware from using efficient vector loads and stores.

In the original implementation, the work is divided into large, contiguous chunks for each program instance (lane). However, because ISPC runs all program instances in a gang concurrently (mapping them to hardware SIMD lanes), the memory access pattern during any single step of the loop looks like this:

* **Lane 0** accesses `input[0]`
* **Lane 1** accesses `input[chunkSize]`
* **Lane 2** accesses `input[2 * chunkSize]`

Because these memory addresses are spread far apart, the CPU cannot load them in a single, fast operation. Instead, the ISPC compiler is forced to emit **gather** instructions to read the data and **scatter** instructions to write it back. Gather/scatter operations are notoriously slow compared to contiguous, aligned vector memory operations.

---

### **The Solution: Interleaved Work Distribution**

To fix this, we need to ensure that the SIMD lanes access adjacent memory locations at the same time. We can achieve this by assigning work in an **interleaved** (or cyclic) pattern.

In this approach, the program instances work together on a contiguous block of `programCount` elements, and then the entire gang strides forward by `programCount`.

Here is the rewritten ISPC code:

```c
void my_ispc_function(uniform int N, uniform float* input, uniform float* output) {
    // Each program instance starts at its own index and strides by the gang size.
    // This ensures that during any single loop iteration, all active instances
    // are accessing adjacent memory locations (i, i+1, i+2, etc.).
    for (int i = programIndex; i < N; i += programCount) {
        output[i] = 2 * input[i];
    }
}

```

### **Why this is better (and "cooler"):**

1. **Contiguous Memory Access:** Lane 0 accesses `input[0]`, Lane 1 accesses `input[1]`, etc. This allows the compiler to generate fast, packed vector load and store instructions.
2. **Built-in Masking:** You don't even need to assume `programCount` divides `N` equally. The loop condition `i < N` acts as an automatic bounds check. On the final iteration of the loop, if `N` is not a perfect multiple of `programCount`, the ISPC compiler will automatically generate a mask to disable the lanes where `i >= N`, safely preventing out-of-bounds memory accesses.

### **1. Cache Behavior Analysis**

To determine the best optimization, we first need to trace the memory accesses and understand the cache hit/miss ratio:

* **Array Size:** The array `A` contains 1024 `float` values. Since a `float` is 4 bytes, the entire array is 4,096 bytes (4 KB).
* **Cache Capacity:** The cache is 32 KB, which is more than large enough to hold the entire 4 KB array at once.
* **Cache Lines:** The cache lines are 4 bytes long, meaning each cache line holds exactly one `float`.
* **First Iteration (i = 0):** During the very first run of the inner loop, the CPU attempts to read `A[0]` through `A[1023]`. Because the cache is empty, every single access results in a **cache miss** (cold misses), incurring the 16-cycle memory latency penalty per access.
* **Subsequent Iterations (i = 1 to 9999):** Because the array easily fits into the 32 KB cache and an LRU policy is used, the entire array remains in the cache after the first outer loop iteration. For the remaining 9,999 iterations, every memory access is a **cache hit**, taking only 1 cycle.

**Conclusion:** The vast majority of memory accesses (99.99%) are cache hits.

---

### **2. Why 32-Way Multi-Threading Fails Here**

Hardware multi-threading is designed to **hide memory latency**. When one thread stalls waiting for a cache miss to fetch data from main memory, the core quickly swaps to another thread to keep the execution units busy.

However, because almost every access in this program is a 1-cycle cache hit, the core is almost never stalled waiting on memory. There is virtually no memory latency to hide. ---

### **3. Why 4-Wide SIMD Succeeds**

SIMD (Single Instruction, Multiple Data) is designed to increase **compute throughput**. It allows the processor to perform the same arithmetic operation on multiple pieces of data simultaneously.

Because the data is already sitting in the cache (meaning it can be fed to the CPU extremely quickly), the bottleneck is no longer memory, but the arithmetic execution itself.

Because the SIMD solution processes 4 array elements per instruction—and the cache is fast enough to feed those execution units without stalling—it cuts the required compute time by a factor of 4. struct Node {
float min, max; // if leaf: start/end of segment, else: bounds on all child segments.
bool leaf; // true if nodes is a leaf node

// -- computes segment id of the largest segment containing points[programIndex]
// -- root_node is the root of the search tree
// -- each program instance processes one query point
export void find_segment_1(uniform float* points, uniform int* results, uniform Node* root_node) {
Stack<Node*> stack;
Node* node;
float max_extent = 0.0;
// p is point this program instance is searching for
float p = points[programIndex];
results[programIndex] = NO_SEGMENT;
stack.push(root_node);
while(!stack.size() == 0) {
node = stack.pop();
while (!node->leaf) {
// [I-test]: test to see if point is contained within this interior node
if (p >= node->min && p <= node->max) {
// [I-hit]: p is within interior node... continue to child nodes
push(node->right);
node = node->left;
} else {
// [I-miss]: point not contained within node, pop the stack
if (stack.size() == 0)
return;
else
node = stack.pop();
}
}
// [S-test]: test if point is within segment, and segment is largest seen so far
if (p >= node->min && p <= node->max && (node->max - node->min) > max_extent) {
// [S-hit]: mark this segment as ‘‘best-so-far’’
results[programIndex] = node->segment_id;
max_extent = node->max - node->min;
}
}
}
export void find_segment_2(uniform float* points, uniform int* results, uniform Node* root_node) {

Node* node;
float max_extent = 0.0;
// p is point this program instance is sarch for
float p = points[programIndex];
results[programIndex] = NO_SEGMENT;

// Top level insertion code (insert val into BST)
bool BST::insert(int val) {
bool result = false;
root.lock();
result = insert_sync(&root, val);
return result;
}
// insertion subroutine
bool BST::insert_sync(Edge *e, int val) {
Node *n = e->get();
if (n == NULL) {
e->set(new Node(val));
e->unlock();
return true;
}
e->unlock();
if (n->value == val) {
return false;
}
Edge *next = (val < n->value) ? &n->left : &n->right;
next->lock();
return insert_sync(next, val);
}
The following is a function for removing the maximum element in the tree. (Notice it always traverses to
the right child until there are no more right children.) It is intended to be thread safe (but may or may
not be).
// Top level remove code. Returns true if a node exists, and fills in val
bool BST::remove_max(int* val) {
root.lock();
return remove_max_sync(root, *val);
}
// removal subroutine
bool BST::remove_max_sync(Edge *e, int *val) {
Node *n = e->get();
if (n == NULL) {
e->unlock();
return false;
}
Edge *next = &n->right;
next->lock();
bool found = remove_max_sync(next, val);
if (!found) {
// Current node holds the maximum value since there
// is no right child
*val = n->value;
// Replace this node with its left subtree
Edge *left = &n->left;
left->lock();
e->set(left->get());
left->unlock();
delete n;
}
e->unlock();
return true;
}
